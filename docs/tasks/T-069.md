- **Depends**: T-017, T-088, T-089

Fixed YAML schema files: renamed `use:` to `mixins:` in all 10 table files, converted function grant `to:` from arrays to strings, added required `to:` field to tenant_scoped policy. ## Description

Replace all hand-written DDL in `packages/db/src/bootstrap.ts` with the `@mabulu-inc/simplicity-schema` programmatic API. The YAML schema files already exist in `packages/db/schema/` (tables, roles, functions, mixins). The task is to wire them up so `bootstrap()` delegates DDL to simplicity-schema instead of executing raw SQL.

## Current state

- `bootstrap(pool, config)` in `packages/db/src/bootstrap.ts` contains ~250 lines of hand-written SQL for creating roles, functions, tables, indexes, triggers, and grants
- `packages/db/schema/` already has YAML definitions for all these objects
- The YAML table files use `use: [timestamps]` but simplicity-schema's parser reads `mixins:` (not `use:`) — these must be renamed

## Key API details

The correct package is `@mabulu-inc/simplicity-schema` (NOT `@mabulu-inc/schema-flow`). The key exports:

```typescript
import { resolveConfig, runPipeline, createLogger, closePool } from '@mabulu-inc/simplicity-schema';
// NOTE: use runPipeline (or runMigrate for migrate-phase only), NOT runAll

const config = resolveConfig({
  connectionString: '...',    // required — the PG connection URL
  baseDir: '...',             // path to schema/ directory (NOT package root)
  pgSchema: '...',            // target PG schema (default: 'public')
  quiet: true,                // suppress output
});

const logger = createLogger({ verbose: false, quiet: true, json: false });

const result = await runPipeline(config, logger);
// result: ExecuteResult { success, operationsExecuted, errors }

// IMPORTANT: release simplicity-schema's internal connection pool after each call
await closePool();
```

**Connection string bridge**: `ProjectConfig.database` contains the connection URL. Pass it as `connectionString` to `resolveConfig()`.

**Schema directory**: The YAML files are in `packages/db/schema/`. Use `path.resolve(__dirname, '..', 'schema')` for `baseDir`. Note: `baseDir` points to the `schema/` directory itself, NOT the package root.

**pgSchema**: Read from `config.systemSchema ?? config.schema ?? 'public'` (same logic as current bootstrap).

## Approach

### Step 1: Add dependency and rename YAML key

1. Add `@mabulu-inc/simplicity-schema` as a dependency in `packages/db/package.json` and run `pnpm install`
2. Rename `use:` → `mixins:` in all YAML table files that reference mixins. The files are:
   - `packages/db/schema/tables/users.yaml`
   - `packages/db/schema/tables/tenants.yaml`
   - `packages/db/schema/tables/memberships.yaml`
   - `packages/db/schema/tables/permission_overrides.yaml`
   - `packages/db/schema/tables/simplicity_dashboards.yaml`
   - `packages/db/schema/tables/simplicity_notification_rules.yaml`
   - `packages/db/schema/tables/simplicity_notifications.yaml`
   - `packages/db/schema/tables/simplicity_state_machines.yaml`
   - `packages/db/schema/tables/simplicity_transition_log.yaml`
   - `packages/db/schema/tables/simplicity_widgets.yaml`
3. Update `packages/db/tests/schema-validation.test.ts` — change `schema.use` → `schema.mixins` on lines 51, 79, and 120

### Step 2: Rewrite bootstrap.ts

Replace hand-written DDL with simplicity-schema calls:
- Call `resolveConfig()` + `runPipeline()` (or `runMigrate()`) + `closePool()`
- Keep `seedDefaults()` as-is (bcrypt hashing can't be done in YAML)
- Wrap simplicity-schema errors in `DatabaseError` with code `DB_005`
- Keep the `bootstrap()` function signature compatible with existing callers (it takes `pool: ConnectionPool` and a config object — the config needs a `database` / `connectionString` field and optional `schema`)

### Step 3: Verify idempotency

The timestamps mixin uses `WHEN (OLD.* IS DISTINCT FROM NEW.*)` on its trigger. Previous versions of simplicity-schema could not introspect this correctly (pg_get_expr limitation), but this was fixed upstream in simplicity-schema T-086. After upgrading the dependency, `runPipeline` should produce zero operations on a second run. **Do not remove the `when` clause from `timestamps.yaml` — it should work as-is.** If idempotency tests fail, ensure you are on the latest version of `@mabulu-inc/simplicity-schema`.

### Step 4: Update provider.ts

Update `packages/db/src/provider.ts` `migrate()` to use simplicity-schema. The provider already captures `connectionUrl` from `connect()`. Use that in `resolveConfig({ connectionString: connectionUrl, ... })`. Do NOT change the `MigrationConfig` type in `@simplicity-admin/core` — it doesn't need `connectionString` because the provider already has it.

### Step 5: Update tests

- `packages/db/tests/bootstrap.test.ts` — ensure tests pass with the new bootstrap. The `bootstrap()` signature needs a `connectionString` or `database` field. If you change the signature, update this test's `beforeAll` and the idempotency test accordingly.
- `packages/db/tests/provider.test.ts` — verify provider tests still pass
- `packages/cli/tests/dev.test.ts` — if bootstrap signature changed, make sure the CLI's `dev` command still calls it correctly. The dev command test starts a server that calls bootstrap; if the connection string isn't plumbed through, you'll get `SASL: client password must be a string`.

### Step 6: Final verification

Run `pnpm check` from the repo root. All tests across all packages must pass, not just `packages/db`.
