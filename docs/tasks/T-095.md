## Description

Developers need to know upfront that SIMPLICITY-ADMIN will create a `_simplicity` schema in their database and populate it with system tables. This should not be a surprise discovered after running `dev` for the first time.

### 1. Prominent callout on Quick Start and Introduction pages

Both the Quick Start and Introduction pages in the docs site must include an early, visible callout (Starlight `:::note` or `:::caution` admonition) that says something like:

> **SIMPLICITY-ADMIN creates a `_simplicity` schema in your database.** On first startup, it creates system tables for users, tenants, memberships, roles, permissions, workflows, dashboards, and notifications. Your application tables in `public` are never modified. You can change the schema name via `systemSchema` in your config.

This should appear before or immediately after the first `pnpm create` / startup instructions — before the user runs anything.

The README.md should also include this callout in the Quick Start section.

### 2. Dedicated "System Schema" reference page

Create `docs-site/src/content/docs/reference/system-schema.md` — a full reference page documenting:

- **When the schema is created** — during `bootstrap()`, which runs on first `dev` startup or `migrate apply`
- **What schema name** — defaults to `_simplicity`, configurable via `systemSchema`
- **Each system table** with:
  - Purpose (one sentence)
  - Column listing (name, type, brief description)
  - Who has access (which roles, what operations)
  - Relationship to other system tables
- **The system tables** (derive from `packages/db/schema/tables/*.yaml`):
  - `users` — user accounts
  - `tenants` — workspace/org isolation
  - `memberships` — user-to-tenant role assignments
  - `revoked_tokens` — JWT blocklist
  - `simplicity_state_machines` — workflow definitions
  - `simplicity_transition_log` — workflow audit trail
  - `simplicity_dashboards` — dashboard configs
  - `simplicity_widgets` — dashboard widget configs
  - `simplicity_notification_rules` — notification triggers
  - `simplicity_notifications` — delivered notifications
  - `simplicity_permission_overrides` — UI permission denials
- **Default seed data** — default tenant ("Default"), default admin user (admin@localhost), default membership
- **Database roles** created — `authenticator`, `anon`, `app_viewer`, `app_editor`, `app_admin`

Add this page to the docs-site sidebar under Reference.

### 3. Update astro.config.mjs sidebar

Add "System Schema" entry to the Reference section in the sidebar.
