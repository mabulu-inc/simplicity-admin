# T-088: Test infrastructure — Docker lifecycle and per-test database isolation

- **Status**: TODO
- **Milestone**: M6 — Code Quality & Hardening
- **PRD Reference**: §6 (NFR-011)
- **Depends**: none

## Description

Tests currently require manually running `docker compose up -d` before `pnpm test`, and test suites share schemas at the describe-block level. This task fixes both:

1. **Automatic Docker lifecycle** — Vitest `globalSetup` starts the Postgres container before any tests run and tears it down after all tests complete. No manual `docker compose up` required.

2. **Per-test database isolation** — Each individual test gets its own database, not just a schema per describe block. Prevents cross-test pollution and enables future parallel execution.

3. **Shared test helper** — A shared helper providing `createTestDb()`/`destroyTestDb()` used in `beforeEach`/`afterEach` across all integration tests.

4. **Environment** — Tests read `DATABASE_URL` from `.env`. The globalSetup ensures the container is healthy before proceeding.

## Produces

- Vitest `globalSetup.ts` — starts/stops Docker container
- Shared test helper (`createTestDb`/`destroyTestDb`)
- Updated all integration tests across packages to use per-test databases
- Updated vitest configs to reference globalSetup

## Hints

- Look at existing test patterns in `packages/db/tests/` and `packages/auth/tests/` for current schema isolation approach
- `docker compose up -d --wait` blocks until healthcheck passes
- `docker compose down -v` removes volumes for clean teardown

- **Tests**: `pnpm test` passes end-to-end without manual Docker setup; no shared mutable state between tests; tests pass regardless of execution order
- **AC**: Running `pnpm test` on a clean machine (with Docker installed) works with zero manual steps; every integration test creates its own database and cleans up after itself
